💡 3. 핵심 비즈니스 가설 3종

PRD v1.1 실험 계획(§8 Exp-1~4), PRD v1.2 AI Reviewer Agent 명세(§2), SRS v1.3 기관 URL 설정(§2), JTBD 인터뷰(source_10) 결과를 종합하여 다음 3개의 핵심 가설을 수립합니다.

가설 1 — AI 판독 추천 UI를 통한 의사결정 시간 단축
🔍 배경 (Problem Context):
현재 인사담당자는 지원자 서류를 수기로 검증할 때, 직접 발급처 사이트(Q-Net, 정부24, YBM 등)에 접속하여 정보를 입력하고 결과 화면을 육안으로 비교하는 데 건당 평균 12분이 소요됩니다. 4,000건 공채 기준 총 2주(약 800시간)가 필요합니다.

💡 가설 (Hypothesis Statement):
AI Reviewer Agent(Gemini Vision LLM)가 원본 서류 이미지와 RPA 캡처본을 3중 교차 검증(Triple Check Loop)하여 도출한 판독 권고(APPROVE / REJECT / ESCALATE)와 한국어 자연어 근거 요약을 대시보드에 제공하면, 담당자의 서류 판정 최종 의사결정 시간이 기존 12분 → 30초 이하 (약 96% 단축)될 것이다.

🧪 실험 설계 (Experiment — PRD §8 Exp-1 연계):
  • 방법: A/B 테스트 — A그룹(AI 판독 추천 + 원클릭 승인), B그룹(AI 없이 원본·캡처 이미지만 직접 검토)
  • 표본: 공공기관 담당자 각 50명 (n=100)
  • 측정 기간: 1개 채용 회차 (약 2~4주)
⏱ 기준선: 건당 12분 (수기) → 목표: p95 ≤ 30초
🤖 AI Override Rate (담당자 뒤집기 비율) ≤ 5%
📊 A그룹 감사 대응 자신감 ≥ B그룹 + 1.0점 (Likert 5점 척도)
가설 2 — Self-Service 소명/재제출 루프를 통한 민원 차단
🔍 배경 (Problem Context):
서류 불일치 혹은 누락 발생 시, 담당자가 지원자에게 일일이 유선이나 개별 메일로 연락하여 보완을 요청하는 민원 전화 대응이 전체 업무의 70~90%를 차지합니다(VPS 5-2, JTBD 인터뷰 — 박지민 유형 3인 증언). 이로 인해 전략적 채용 업무에 할애할 시간이 극단적으로 부족합니다.

💡 가설 (Hypothesis Statement):
검증 결과 반려(REJECT) 판정 시, 구체적인 불일치 사유(어떤 서류의 어떤 항목이 불일치), 필요한 보완 서류 안내, 그리고 72시간 유효 재제출 링크를 포함한 알림 메일을 Resend API로 자동 발송하는 Self-Service 루프를 가동하면, 담당자의 수동 개입(전화·메일) 건수가 90% 이상 감소할 것이다.

🧪 실험 설계 (Experiment — PRD §8 Exp-2 연계):
  • 방법: Before-After 비교 — 도입 전 채용 회차 vs 도입 후 채용 회차 (동일 기관, 동일 규모)
  • 측정 지표: 회차당 담당자 수동 전화 건수, Self-Service 처리율(%), 재제출 완료 소요 시간(시간)
  • 측정 도구: 시스템 이벤트 로그 (알림 발송 → 재제출 완료 펀넬)
📧 기준선: 민원 전화가 담당자 업무의 70~90% 차지
🎯 목표: Self-Service 처리율 ≥ 90% (수동 개입 ≤ 10%)
🔗 재제출 링크 보완 완료율 ≥ 85% (72시간 이내)
📉 수동 개입 건수 ≥ 90% 감소
가설 3 — 비개발자 PM의 기관 설정 hot-load 완결성
🔍 배경 (Problem Context):
졸업증명서 발급처만 해도 웹민원센터, 써트피아, 정부24, 아이써티 등 4개 이상이며, 아이써티는 학교마다 진위확인 URL이 모두 다릅니다(SRS v1.3 §2). 새로운 사이트가 추가되거나 기존 사이트의 HTML 구조가 바뀔 때마다 개발자가 스크립트를 수정하고 배포하면, 대응까지 1~3일 소요되어 그 사이 검증이 중단됩니다.

💡 가설 (Hypothesis Statement):
config/agency_config.json hot-load 엔진(재시작 없이 실시간 반영)과 대시보드 UI 상의 AI 기반 CSS 셀렉터 자동 추천 기능(Gemini가 HTML 소스를 분석하여 입력 필드 후보군 제안)을 제공하면, 비개발자 PM이 개발자 지원 없이 3분 이내에 신규 진위확인 사이트 설정을 완료하고 RPA 구동 테스트까지 성공할 수 있을 것이다.

🧪 실험 설계:
  • 방법: 과업 완료 테스트(Task Completion Test) — PM 5인에게 미등록 대학 사이트 URL을 주고, 설정 UI만으로 등록~RPA 테스트 성공까지 소요 시간 측정
  • 측정 지표: 과업 완료 시간(분), AI 셀렉터 추천 성공률(%), 개발자 호출 여부
  • 표본 규모: PM 5인 × 대학 3곳 = 15회 시행
⚙️ 기준선: 개발자 의뢰 기반 기관 추가 1~3일
🎯 목표: PM 직접 설정~RPA 테스트 완료 ≤ 3분
🤖 AI CSS 셀렉터 추천 성공률 ≥ 70%
🚫 개발자 호출 빈도 0회 (완전 자립)
